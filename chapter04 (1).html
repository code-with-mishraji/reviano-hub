Pillar 1: HTML Tables Fundamentals & Semantic Structure 📊

Basic Structure — Table Ke 4 Core Tags

Table banane ke liye 4 tags ek saath kaam karte hain:

<table>
    <tr>
        <th>Name</th>
        <th>Age</th>
    </tr>
    <tr>
        <td>Rohan</td>
        <td>22</td>
    </tr>
    <tr>
        <td>Priya</td>
        <td>24</td>
    </tr>
</table>
  • <table> — Ye poori table ka **parent container** hai। Iske andar hi saara table-related content aata hai।
  • <tr> (Table Row) — Ek **horizontal row** banata hai। Har naya <tr> matlab ek naya row।
  • <td> (Table Data) — Ek **normal cell** banata hai jisme actual data hota hai (jaise "Rohan", "22")।
  • <th> (Table Header) — Ek **header cell** banata hai (jaise column ka naam "Name", "Age")।

<th> automatically bold aur centered kyun dikhta hai?

Har browser ke paas ek **default built-in stylesheet** hoti hai (jise "User Agent Stylesheet" kehte hain), jisme <th> ke liye already ye rules likhe hue hain:

th {
    font-weight: bold;
    text-align: center;
}

Isliye <th> automatically bold aur center-aligned dikhta hai, bina tumhare kuch CSS likhe hue — kyunki browser semantically samajhta hai ki ye ek "header" hai, normal data nahi, isliye use visually distinguish karta hai।

🧠 Real-world Analogy: HTML Table bilkul **"Microsoft Excel Sheet"** jaisa concept hai! <table> poori spreadsheet hai, <tr> har horizontal row hai (Row 1, Row 2, Row 3...), aur <td>/<th> individual cells hain jaise Excel mein A1, B1, C1। Aur <th> exactly waisa hai jaise Excel mein tum apni **header row ko bold aur centered** karte ho taaki data se alag dikhe।

⚠️ Common Mistake: Beginners kabhi kabhi <td> ko har jagah use karte hain, header row ke liye bhi — aur phir manually CSS se bold/center karte hain। Ye **semantically galat** hai! Agar koi cell **header** hai (matlab column/row ka naam batati hai), toh <th> use karo — isse screen readers ko pata chalta hai ki ye ek header hai, aur tumhe manual styling ki zaroorat bhi nahi padegi।

Semantic Tables — <thead>, <tbody>, <tfoot>

Chhoti tables ke liye seedha <tr> likhna theek hai, lekin **professional, production-level tables** mein content ko 3 semantic groups mein baanta jaata hai:

<table>
    <thead>
        <tr>
            <th>Product</th>
            <th>Price</th>
        </tr>
    </thead>

    <tbody>
        <tr>
            <td>Laptop</td>
            <td>₹50,000</td>
        </tr>
        <tr>
            <td>Mouse</td>
            <td>₹500</td>
        </tr>
    </tbody>

    <tfoot>
        <tr>
            <td>Total</td>
            <td>₹50,500</td>
        </tr>
    </tfoot>
</table>
  • <thead> — Table ka **header section** group karta hai (column names)।
  • <tbody> — Table ka **main data (body)** group karta hai — actual rows of data।
  • <tfoot> — Table ka **footer section** group karta hai (jaise totals, summaries)।

Ye kyun use karte hain? 3 Bade Fayde:

  1. SEO: Search engines (Google) table ke structure ko better samajh paate hain jab ye semantic groups clearly diye gaye hon — isse Google ko pata chalta hai ki kaunsa data "header info" hai aur kaunsa "actual content", jo rich search results (jaise table snippets) mein help karta hai।
  2. Accessibility (Screen Readers): Jab koi visually-impaired user screen reader se table navigate karta hai, toh <thead> ki wajah se screen reader **"header row"** announce kar sakta hai jab user kisi bhi row par focus kare — jisse user ko baar-baar samajhna nahi padta ki kaunsa column kya represent karta hai।
  3. Printing Long Tables: Agar tumhari table bahut lambi hai aur multiple pages mein print ho rahi hai, toh browsers <thead> ko **har print page ke top par automatically repeat** kar sakte hain! Isse har printed page par header row dikhती rehti hai, chahe woh page 1 ho ya page 5 — user ko baar-baar upar scroll karke column names dekhne ki zaroorat nahi padti।

🧠 Real-world Analogy: Socho ek **lambi Excel report** jise tum print karte ho — agar report 10 pages ki hai, toh smart Excel settings har page ke top par **column headers repeat** karti hain, taaki page 7 dekhte waqt bhi tumhe pata rahe ki kaunsa column "Price" hai aur kaunsa "Quantity"। <thead> exactly yehi kaam karta hai HTML tables mein!

💡 Pro-Tip: Even chhoti tables mein bhi <thead>/<tbody> use karna ek **good habit** hai, kyunki ye tumhare CSS ko bhi organize karne mein help karta hai — jaise tum easily tbody tr:nth-child(even) jaisा selector use karke alternate rows ko different color de sakte ho (zebra striping), jo <tbody> ke bina thoda mushkil ho jaata hai।

The <caption> Tag — Table Ka Title

Agar tumhe table ka ek proper **title/description** dena ho (jaise "Table 1: Monthly Sales Report"), toh <caption> tag use karo:

<table>
    <caption>Monthly Sales Report — 2026</caption>
    <thead>
        <tr>
            <th>Month</th>
            <th>Revenue</th>
        </tr>
    </thead>
    <tbody>
        <tr>
            <td>January</td>
            <td>₹2,00,000</td>
        </tr>
    </tbody>
</table>

Correct Placement Rule: <caption> ko hamesha <table> tag ke **immediately baad, sabse pehli child** ki tarah likhna chahiye — <thead> se bhi pehle! Ye HTML specification ka strict rule hai। Agar tum ise kahi aur likhoge (jaise <tbody> ke andar ya baad mein), toh browser use ignore ya galat render kar sakta hai।

By default, <caption> table ke **top mein center-aligned** dikhta hai, aur ye screen readers ke liye bhi bahut useful hai — jab koi user table ko navigate karta hai, screen reader pehle caption announce karta hai, jisse user ko pata chal jaata hai ki ye table **kis baare mein** hai।

⚠️ Common Mistake: Log <caption> ki jagah table ke upar ek normal <h3> ya <p> likh dete hain। Ye kaam toh "visually" kar deta hai, lekin **semantic connection** miss ho jaata hai — screen readers ko pata nahi chalega ki ye heading specifically **isी table** ke liye hai। <caption> table ke andar hone ki wajah se programmatically table se juda hua rehta hai।

Pillar 2: Complex Table Layouts (Grid Merging) 🧩

Kabhi tumne dekha hoga ki table mein koi cell **normal se bada** hota hai — jaise ek cell 2 columns ki jagah le raha ho, ya 2 rows ki height cover kar raha ho। Ye colspan aur rowspan attributes se hota hai।

The colspan Attribute — Columns Ko Horizontally Merge Karna

colspan ek cell ko **multiple columns tak horizontally phaila** deta hai। Isse samajhne ke liye ek grid example lete hain:

<table border="1">
    <tr>
        <th colspan="2">Contact Info</th>
    </tr>
    <tr>
        <td>Email</td>
        <td>Phone</td>
    </tr>
</table>

Iska Visual Grid Kaisa Banega?

Row 1:  [   Contact Info (spans 2 cols)   ]
Row 2:  [    Email     ]  [    Phone      ]

Yahan pehli row mein sirf **ek** <th> likha gaya hai, lekin uska colspan="2" hone ki wajah se woh **2 columns ki poori width** le lega — jitni width Row 2 ke dono cells (Email + Phone) milke lete hain। Isliye Row 1 mein tumhe **doosra <th> likhne ki zaroorat nahi** — browser automatically samajh jaata hai ki pehla wala hi 2 columns cover kar raha hai।

The rowspan Attribute — Rows Ko Vertically Merge Karna

rowspan ek cell ko **multiple rows tak vertically phaila** deta hai:

<table border="1">
    <tr>
        <th>Name</th>
        <th>Department</th>
    </tr>
    <tr>
        <td rowspan="2">Rohan</td>
        <td>Engineering</td>
    </tr>
    <tr>
        <td>Design</td>
    </tr>
</table>

Iska Visual Grid Kaisa Banega?

Row 1:  [   Name   ]  [  Department   ]
Row 2:  [          ]  [ Engineering   ]
        [  Rohan   ]
        [(spans 2  ]  [   Design      ]
        [  rows)   ]
Row 3:  [          ]

Yahan "Rohan" wala cell Row 2 aur Row 3 dono ki height cover kar leta hai। Isliye Row 3 mein tumhe **"Name" column ke liye naya <td> likhne ki zaroorat nahi** — sirf "Design" wala <td> likha gaya hai, jo automatically Row 3 ke second column mein slot ho jaata hai।

Browser Rendering Ka Math & Logic — Kaise Push Hote Hain Baaki Cells?

Ye samajhna **bahut zaroori** hai: Jab browser HTML parse karta hai, toh ye ek **invisible grid** ko row-by-row, left-to-right fill karta hai।

  • Jab browser ek colspan="2" wala cell dekhta hai, woh us row mein **2 grid-slots occupy** kar leta hai (na ki 1)। Isliye us row mein agla <td> automatically 2 slots aage se start hota hai।
  • Jab browser ek rowspan="2" wala cell dekhta hai, woh **current row + next row**, dono mein us column-position ko "already filled" maan leta hai। Isliye next row mein jab browser naye <td> ko place karta hai, woh us occupied slot ko **automatically skip** karke aage wale available slot mein place karta hai।

Isliye **golden rule** yaad rakho: Jab kisi cell mein rowspan lagate ho, toh **next row(s) mein us column ke liye ek kam <td> likhna padega** (kyunki wo slot already "occupied" hai upar wale rowspan cell se)।

🧠 Real-world Analogy: Socho ek **Movie Theatre Seating Chart** — normally har seat ek insaan ke liye hoti hai (1 cell = 1 seat)। Lekin agar koi group VIP couple-seats book karte hain jo 2 normal seats jitni jagah leti hain (colspan jaisा), toh us row mein baaki logon ko 2 seats "skip" karke apni seat number find karni padegi। Rowspan waisा hai jaise ek "Reclining Sofa Seat" jo neeche wali row ki jagah bhi cover kar leta hai — us jagah neeche koi doosri seat book nahi ho sakti।

⚠️ Common Mistake: Sabse common aur frustrating mistake hai jab developers colspan/rowspan use karte waqt **grid ka math galat calculate** karte hain — matlab total columns/rows count match nahi hoti। Chalo ek broken example dekhते hain:

<!-- ❌ GALAT: Grid Math Match Nahi Ho Rahi -->
<table border="1">
    <tr>
        <th colspan="2">Header</th>
    </tr>
    <tr>
        <td>A</td>
        <td>B</td>
        <td>C</td>  <!-- Extra cell! Header sirf 2 columns cover karta hai, lekin yahan 3 cells hain -->
    </tr>
</table>

Yahan Row 1 sirf **2 columns** ki width define kar rahi hai (colspan="2"), lekin Row 2 mein **3 cells** (A, B, C) hain! Isse table ka structure **misaligned/broken** ho jaayega — columns properly align nahi honge, aur visually table "tuti-phuti" dikhegi, kyunki browser confuse ho jaata hai ki actual grid width kya honi chahiye।

💡 Pro-Tip: Complex tables banane se pehle, hamesha **pehle paper par ya mentally grid draw karo** — kitne total columns hain, kaunse cells merge honge — phir code likho। Isse colspan/rowspan ki galtiyan bahut kam ho jaati hain। Browser DevTools mein table ko inspect karke bhi tum dekh sakte ho ki actual rendered grid kaisa bana hai।

Pillar 3: The Video Element (<video>) & Deep Attributes 🎬

Basic Setup — <source> Tag Kyun Better Hai Inline src Se

Video embed karne ke 2 tarike hain:

<!-- Method 1: Inline src (Simple, lekin limited) -->
<video src="movie.mp4" controls></video>

<!-- Method 2: <source> tags (Recommended, Production-Ready) -->
<video controls>
    <source src="movie.mp4" type="video/mp4">
    <source src="movie.webm" type="video/webm">
    <source src="movie.ogg" type="video/ogg">
    Your browser does not support the video tag.
</video>

<source> Method Kyun Better Hai?

Problem ye hai ki **har browser har video format support nahi karta** — kuch purane browsers .mp4 support karte hain lekin .webm nahi, ya vice-versa। Agar tum sirf ek inline src doge (jaise sirf .mp4), aur user ka browser use support nahi karta, toh video **bilkul nahi chalega**।

<source> tags ka use karke tum **multiple format options** de sakte ho। Browser top se bottom **saare <source> tags check karta hai**, aur jo **pehla compatible format** milta hai, wahi use karta hai। Isse cross-browser compatibility guarantee hoti hai — agar pehla format kaam nahi karta, browser automatically doosra try karega!

🧠 Real-world Analogy: Ye ek "Multi-Language Menu" jaisा hai jo restaurant customer ko deta hai — Menu mein English, Hindi, aur Regional language, teeno mein items likhe hote hain। Customer (browser) apni **samajh ke hisaab se** jo language usse aati hai, wahi read karta hai। Agar sirf ek language hoti (sirf ek src), toh jo customer wo language nahi samajhta, use kuch samajh hi nahi aata।

Core Attributes Deep-Dive

controls — Ye attribute video par **default browser controls** dikhata hai (play/pause button, volume slider, progress bar, fullscreen button)। Bina isके, video dikhta toh hai lekin user ke paas use control karne ka koi built-in tarika nahi hota (custom JS controls banane hon toh alag baat hai)।

<video controls>...</video>

autoplay — Ye video ko page load hote hi **automatically play** karne ki koshish karta hai।

<video autoplay>...</video>

⚠️ Bahut Zaroori Baat: Modern browsers (Chrome, Firefox, Safari) ne **autoplay policy** implement ki hai — agar video mein **audio hai**, toh autoplay tab tak kaam nahi karega jab tak tum saath mein muted attribute bhi na do! Ye policy isliye banayi gayi kyunki users ko random websites par achanak loud sound se annoy hone se bachana tha।

<!-- ❌ Ye kaam nahi karega (audio ke saath autoplay blocked hoga) -->
<video autoplay controls>...</video>

<!-- ✅ Ye kaam karega (muted hone ki wajah se autoplay allowed hai) -->
<video autoplay muted controls>...</video>

loop — Video khatam hone ke baad **automatically restart** ho jaata hai, continuously repeat karta rehta hai।

<video loop>...</video>

poster — Video load hone se pehle (ya play hone se pehle) jo **thumbnail image** dikhani hai, use set karta hai:

<video controls poster="thumbnail.jpg">
    <source src="movie.mp4" type="video/mp4">
</video>

Bina poster ke, browser generally video ka **pehla frame** dikhata hai (ya ek blank black screen), jo hamesha visually appealing nahi hota। poster attribute se tum ek professionally designed thumbnail (jisme play-button ka overlay bhi ho sakta hai) set kar sakte ho।

preload — Ye batata hai ki page load hote waqt video data **kitna pehle se download** kiya jaaye, chahe user ne play button abhi tak click na kiya ho:

ValueKya Karta Hai
noneVideo ka koi bhi data preload nahi hota — sirf jab user play dabaye, tabhi loading start hoti hai। Data-saving ke liye best (jaise mobile users ke liye jinke paas limited data plan hai)।
metadataSirf video ki basic information (duration, dimensions, pehla frame) preload hoti hai, actual video content nahi। Ye default-ish balanced choice hai।
autoBrowser ko poori freedom hai — woh poora video (ya jitna zyada se zyada possible ho) preload kar sakta hai, taaki jab user play dabaye, video instantly bina buffering ke chale।
<video controls preload="metadata">...</video>

💡 Pro-Tip: Agar tumhari website mein bahut saari videos hain (jaise ek video-gallery page), toh preload="none" use karo taaki page load fast rahe aur users ka data bhi bache — user jab specific video par click karega, tabhi uski loading start hogi।

Video Subtitles — The <track> Tag

Agar tumhe apni video mein **subtitles ya captions** add karni hain (jo accessibility ke liye bhi bahut important hai — deaf/hard-of-hearing users ke liye, aur different language speakers ke liye), toh <track> tag use karo:

<video controls>
    <source src="movie.mp4" type="video/mp4">
    <track 
        kind="subtitles" 
        src="subtitles-en.vtt" 
        srclang="en" 
        label="English" 
        default>
    <track 
        kind="subtitles" 
        src="subtitles-hi.vtt" 
        srclang="hi" 
        label="Hindi">
</video>
  • kind="subtitles" — Batata hai ki ye track "subtitles" ke liye hai (doosri values bhi hoti hain jaise captions, descriptions, chapters)।
  • src="subtitles-en.vtt" — Woh file jisme actual subtitle text aur timing information hoti hai (.vtt format mein)।
  • srclang="en" — Batata hai ye subtitles kis language mein hain (English)।
  • label="English" — User ko dikhने wala friendly naam jab woh subtitle options mein se choose karega।
  • default — Ye batata hai ki agar multiple <track> hon, toh **kaunsa track by default active** rahega (jab tak user manually na badle)।

.vtt Files Kaise Kaam Karti Hain?

.vtt (WebVTT — Web Video Text Tracks) ek simple text-format file hai jisme timestamps ke saath text likha jaata hai:

WEBVTT

00:00:01.000 --> 00:00:04.000
Namaste! Welcome to Reviano Hub.

00:00:05.000 --> 00:00:08.000
Aaj hum HTML Tables seekhenge.

Browser is file ko parse karta hai aur video play hote waqt exact timestamp par exact text ko subtitle ki tarah screen par overlay kar deta hai।

🧠 Real-world Analogy: <track> tag ek **"Movie Theatre ke Subtitle Screen"** jaisा hai jo kuch international film festivals mein hota hai — movie apni original language mein chalti hai, lekin neeche ek alag screen par timed translation dikhti rehti hai। .vtt file wo "script" hai jo batati hai kaunsa text kis exact second par dikhna chahiye।

⚠️ Common Mistake: Bahut developers autoplay lagane ke baad bhool jaate hain ki mobile devices (especially iOS Safari) mein autoplay policies **desktop se bhi zyada strict** hain — kabhi kabhi muted ke saath bhi autoplay work nahi karta jab tak playsinline attribute bhi na diya jaaye (taaki video inline play ho, automatically fullscreen na ho jaaye)। Mobile-heavy websites banate waqt hamesha real devices par test karo!

Pillar 4: The Audio Element (<audio>) & Web Formats 🎵

Basic Structure & Core Attributes

<audio> tag <video> jaisा hi kaam karta hai, bas isme visual video frame nahi hota — sirf audio player controls dikhte hain:

<audio controls>
    <source src="song.mp3" type="audio/mpeg">
</audio>

Core attributes bhi almost same hain jaise video mein:

  • controls — Play/pause button, volume, progress bar dikhata hai।
  • autoplay — Audio ko page load hote hi play karne ki koshish karta hai (yahan bhi modern browsers strict policies follow karte hain — bina user interaction ke autoplay generally block ho jaata hai, chahe muted ho ya na ho, kyunki audio-only content ke liye policies aur bhi strict hain)।
  • loop — Audio khatam hone ke baad automatically restart hota hai।
  • muted — Audio ko by default **mute** kar deta hai (user ko manually unmute karna padega)।
<audio controls loop muted>...</audio>

Browser Fallbacks — Multiple Formats (MP3, OGG, WAV)

Jaise video mein, audio mein bhi different browsers different formats support karte hain। Isliye multiple <source> dena best practice hai:

<audio controls>
    <source src="podcast.mp3" type="audio/mpeg">
    <source src="podcast.ogg" type="audio/ogg">
    <source src="podcast.wav" type="audio/wav">
    Your browser does not support the audio element.
</audio>
  • MP3 (audio/mpeg): Sabse universally supported format — almost har modern browser isko support karta hai। Compressed, chhoti file size, decent quality — general web use ke liye best choice।
  • OGG (audio/ogg): Open-source format, kuch purane Firefox/Chrome versions ke liye fallback ke taur par use hota tha jab MP3 licensing issues the (aajkal MP3 patent-free hai, isliye OGG ki zaroorat kam ho gayi hai, lekin fir bhi legacy support ke liye rakha jaata hai)।
  • WAV (audio/wav): Uncompressed, **bahut high quality** lekin **bahut badi file size**। General web streaming ke liye impractical hai (slow loading), lekin professional audio-editing tools ya short sound-effects ke liye use hota hai।

Fallback Text Kaise Kaam Karta Hai?

<audio> tag ke andar jo text tum <source> tags ke **baad** likhते ho, wo un **"ancient" browsers** ke liye hai jo <audio> tag ko bilkul samajhते hi nahi (jaise Internet Explorer 8 aur usse purane versions)। Aise browsers <audio> tag ko completely ignore kar dete hain aur seedha uske andar likha gaya **plain text content** dikha dete hain — jisse user ko pata chal jaata hai ki uska browser outdated hai।

🧠 Real-world Analogy: Multiple audio formats dena aisा hai jaise koi restaurant apna menu **3 different formats** mein print kare — QR code (digital, modern), printed paper menu (traditional), aur agar dono na chalein, toh waiter verbally bata de (fallback text)। Har customer (browser) apni capability ke hisaab se **best available option** use karta hai।

💡 Pro-Tip: Aaj ke modern web mein (2026 tak), sirf **MP3** dena bhi generally kaafi hota hai kyunki isकी support almost universal hai। Lekin agar tum **maximum compatibility** chahte ho (jaise old enterprise systems ya legacy browsers support karna ho), toh MP3 + OGG combo safe choice hai।

Pillar 5: Modern Images & Responsive Media 🖼️

<img> Tag Recap — Essential Attributes

<img src="cat.jpg" alt="A fluffy orange cat sitting on a windowsill" width="400" height="300">
  • src — Image file ka path/URL।
  • alt — **Alternative text** jo image ko describe karta hai।
  • width / height — Image ka display size pixels mein।

alt Attribute Ki Importance — SEO & Accessibility Dono Ke Liye:

  1. Accessibility: Screen readers (visually-impaired users ke liye) alt text ko **loudly read out** karte hain, taaki blind users bhi samajh sakein image mein kya hai। Bina alt ke, screen reader sirf "image" bol dega, jo koi useful information nahi deta।
  2. SEO: Google **images ko "dekh" nahi sakta** jaise humans dekhte hain — woh alt text padh ke samajhta hai image kis baare mein hai। Isse tumhari images **Google Image Search** mein rank kar sakti hain, jo extra traffic laa sakta hai।
  3. Broken Image Fallback: Agar kisi wajah se image load nahi ho paati (galat path, slow internet, server down), toh browser alt text ko image ki jagah **display** kar deta hai, taaki user ko pata chale wahan kya hona chahiye tha।

⚠️ Common Mistake: Log alt="" khali chhod dete hain ya keyword-stuffing karte hain (jaise alt="cat cat cats cute cat buy cat food" — SEO manipulate karne ki koshish)। Ye dono galat practices hain — alt text **natural, descriptive, aur genuine** hona chahiye, jaise alt="An orange tabby cat stretching on a wooden windowsill"।

Performance Optimization — loading="lazy"

Agar tumhari webpage mein **bahut saari images** hain (jaise ek e-commerce product listing page, ya photo gallery), toh sab images ko **ek saath load** karna page ko slow bana deta hai — chahe user ne scroll karke unhe dekha bhi na ho!

<img src="product.jpg" alt="Product photo" loading="lazy">

loading="lazy" browser ko batata hai: "Is image ko **turant load mat karo** — sirf tab load karo jab user scroll karke is image ke **near** pahunch jaaye।" Isse initial page load bahut fast hota hai, kyunki browser sirf woh images pehle load karta hai jo **immediately visible** hain (viewport ke andar), aur baaki images "lazy" (der se, jab zaroorat ho) load hoti hain।

🧠 Real-world Analogy: Lazy Loading ek **"Buffet System"** jaisा hai jaha waiters saara khaana pehle se plate mein nahi sajate — jab tum ek dish ke paas pahunchte ho, tabhi woh dish serve hoti hai। Isse kitchen (server/browser) ka load kam hota hai, aur tumhe bhi turant relevant khaana milta hai bina lamba wait kiye pura buffet spread hone ka।

💡 Pro-Tip: loading="lazy" sirf un images par lagao jo **page ke neeche** (fold ke baad) hain। Website ke **sabse upar** wali important image (jaise hero banner) ko lazy-load mat karo — usko turant load hona chahiye kyunki user page open karte hi use dekhega। Isके liye loading="eager" (default behavior) rakho ya attribute hi mat likho।

The <picture> Element — Modern Responsive Images

Ab tak humne ek hi image ko "chhota-bada" dikhane ki baat ki hai (jo CSS se hoti hai)। Lekin kabhi tumhe chahiye hota hai ki **completely different image** dikhaye jaaye based on screen size — isko **"Art Direction Problem"** kehte hain। Jaise, mobile par ek **zoomed-in, cropped** version ho, aur desktop par **wide, full-scene** version ho।

<picture> element ye problem **bina JavaScript ke** solve karta hai:

<picture>
    <source media="(min-width: 1024px)" srcset="banner-desktop.jpg">
    <source media="(min-width: 600px)" srcset="banner-tablet.jpg">
    <img src="banner-mobile.jpg" alt="Reviano Hub Banner">
</picture>

Ye Kaise Kaam Karta Hai?

  • Browser **top se bottom** har <source> ki media condition ko check karta hai।
  • Jaise hi koi condition **match** karti hai (jaise screen width 1024px ya usse zyada hai), browser **usी <source> ka srcset** use karke image load karta hai, aur baaki sources ko ignore kar deta hai।
  • Agar **koi bhi** <source> match nahi karta, toh browser **fallback** ki tarah andar wale <img> tag ka src use karta hai। Isliye <img> tag hamesha **sabse aakhir mein** likhna zaroori hai — ye **mandatory fallback** hai, kabhi bhi <picture> ke andar sirf <source> tags nahi ho sakte, ek <img> hona hi chahiye।

🧠 Real-world Analogy: Socho tumhare paas ek **hi painting ke 3 alag versions** hain — ek bahut bade canvas par (wide detailed view), ek medium canvas par, aur ek chhote canvas par (sirf main subject, zoomed-in)। Depending on **room ka size** (screen width), tum decide karte ho **kaunsa canvas size fit hoga** — bade hall mein bada canvas achha lagega, lekin chhoti room mein wahi bada canvas overwhelming/wasteful hoga, wahan chhota, focused canvas better hai। <picture> element exactly yehi decision automatically browser ko lene deta hai!

💡 Pro-Tip: <picture> element sirf "art direction" (different crop/composition) ke liye hai। Agar tumhe sirf **same image ke different resolutions** deni hain (jaise high-res retina screens ke liye sharper image, bina composition change kiye), toh iske liye <img> tag ka **srcset aur sizes attribute** zyada appropriate hota hai — ye ek related lekin thoda different concept hai jise hum future chapters mein deeply explore karenge!

⚠️ Common Mistake: Beginners <picture> element ke andar <img> tag ko **bhool jaate hain**, ya use bhi <source> jaisा treat karte hain। Yaad rakho — <picture> ke andar sirf **ek hi <img>** hoga (jo fallback ka kaam karta hai), aur woh hamesha **sabse aakhir** mein hoga, chahe kitne bhi <source> tags ho।

🎓 Chapter Summary — Ek Nazar Mein

  1. Tables ke core tags <table>, <tr>, <td>, <th> hain — aur professional tables mein <thead>, <tbody>, <tfoot> use karo SEO, accessibility, aur print-friendliness ke liye, saath mein <caption> se table ka title do।
  2. colspan columns ko horizontally merge karta hai, rowspan rows ko vertically merge karta hai — dono ke count grid ke math se exactly match hone chahiye, warna table misaligned ho jaayegi।
  3. <video> tag mein <source> tags multiple format fallback dete hain — controls, loop, poster, aur preload attributes deep customization dete hain, aur autoplay ke liye muted mandatory hai।
  4. <audio> tag video jaisा hi hai bas visual frame ke bina — MP3/OGG/WAV multiple sources compatibility ensure karte hain।
  5. Images mein alt text SEO/accessibility ke liye critical hai, loading="lazy" performance boost deta hai, aur <picture> element art-direction problem ko bina JavaScript ke solve karta hai।
